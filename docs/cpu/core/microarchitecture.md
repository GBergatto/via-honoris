# Microarchitecture

The microarchitecture of the 5-stage RISC-V core is detailed in the diagram
below. Module and signal names are kept consistent with the SystemVerilog
source files for direct cross-referencing.

![Architecture diagram](https://dummyimage.com/800x400/eee/aaa){ align=left }

**Note:** Signals throughout the core follow a standard naming convention where
an `_X` suffix denotes the corresponding pipeline stage: **F**etch, **D**ecode,
**E**xecute, **M**emory, or **W**riteback.

The following sections will cover the implementation details of the various
modules, assuming a foundational understanding of computer architecture. For a
comprehensive introduction to these concepts applied to the RISC-V ISA, the
following references are recommended:

- "Digital Design and Computer Architecture" (RISC-V edition) by S. Harris and D. Harris
- "Computer Organization and Design" (RISC-V edition) by D. Patterson and J. Hennessy

## Register file

The [register file] contains 32 general-purpose 32-bit registers, with `x0`
hardwired to zero.

[register file]: https://github.com/GBergatto/via-honoris/blob/main/core/rtl/regfile.sv

To minimize data hazards within the pipeline, register writes must effectively
occur in the first half of the clock cycle and reads in the second half. This
timing ensures that a value written by the Writeback stage can be read by the
Decode stage within the same cycle. This behavior is implemented using an
**internal bypass**: if the current write destination (`rd`) matches either
requested source register (`rs1` or `rs2`), the incoming `rd_data` is
combinationally routed directly to the corresponding output (`rs1_data` or
`rs2_data`).

## Immediate generator

The [immediate generator] extracts the immediate values encoded in different bit
fields of the instruction word based on its type (I, S, B, J, or U-type) and
reassembles them into a 32-bit operand.

[immediate generator]: https://github.com/GBergatto/via-honoris/blob/main/core/rtl/immgen.sv

## Control unit

The [control unit] decodes the instructions and generates the control signals
required to coordinate the datapath. It interprets the `opcode` and `funct3`
fields to manage execution flow, register and memory access, and the selection
of ALU operand sources.

[control unit]: https://github.com/GBergatto/via-honoris/blob/main/core/rtl/control.sv

## Arithmetic Logic Unit (ALU)

The [ALU module] performs the requested operation on operands `op1` and `op2`,
returning the result on the `out` port. The operand inputs are treated as
unsigned unless specifically required by the operation (such as arithmetic
right shift or signed comparison). The module also generates `zero`, `lt`
(signed less-than), and `ltu` (unsigned less-than) flags, used to evaluate
conditional branches.

[ALU module]: https://github.com/GBergatto/via-honoris/blob/main/core/rtl/alu.sv

The specific operation is selected via the `alu_op` signal. This signal is
generated by the [ALU control module] based on the `opcode`, `funct3`, and
`funct7` fields of the current instruction.

[ALU control module]: https://github.com/GBergatto/via-honoris/blob/main/core/rtl/alu_control.sv

## Memory interface

The [hp_core] module exposes separate instruction (`imem`) and data (`dmem`)
memory interfaces. This dual-port design allows the Fetch and Memory pipeline
stages to operate concurrently without structural hazards. Externally, both
ports connect to a Wishbone bus and route to the same system memory (currently
implemented as FPGA Block RAM). This maintains a unified address space for both
code and data, as required by the RISC-V specification. Memory-mapped I/O
(MMIO) access is also routed through the `dmem` interface. For more details,
refer to the [SoC architecture] page.

[hp_core]: https://github.com/GBergatto/via-honoris/blob/main/core/rtl/hp_core.sv
[SoC architecture]: ../soc/sys-architecture/

Internally, the core simplifies the external memory adapter logic by natively
handling sub-word loads and stores. It generates the appropriate byte-select
masks (`dmem_we`) for store alignment and performs sign-extension for loads
directly within the pipeline. To synchronize with the bus, the core relies on a
single `ext_stall` input. This signal allows the external memory subsystem to
freeze the entire pipeline during multi-cycle MMIO transactions or when
instruction fetches contend with data memory operations.

## CSR file

The [Control and Status Register file] implements the special-purpose registers
required to track execution state, manage privilege modes, and handle traps.
The RISC-V privileged architecture allocates a 12-bit encoding space for up to
4096 CSRs. By convention, the upper four bits of this address inherently encode
the register's read/write accessibility and the minimum privilege level
required to access it.

[Control and Status Register file]: https://github.com/GBergatto/via-honoris/blob/main/core/rtl/csr_file.sv

The following table identifies the CSRs actively integrated into the core's
hardware logic.

|**Address**|**Register**|**Description**|
|---|---|---|
|**0x300**|`mstatus`|**Machine Status**: global tracking of current privilege mode and global interrupt enable bits.|
|**0x304**|`mie`|**Machine Interrupt Enable**: bitmask to selectively enable timer, software, or external interrupts.|
|**0x305**|`mtvec`|**Machine Trap-Vector Base-Address**: trap handler base address and interrupt vectoring mode.|
|**0x340**|`mscratch`|**Machine Scratch**: scratchpad for the trap handler to swap context pointers.|
|**0x341**|`mepc`|**Machine Exception PC**: automatically captures the address of the interrupted instruction when a trap occurs.|
|**0x342**|`mcause`|**Machine Cause**: automatically updates upon a trap with the exception or interrupt code (e.g., timer interrupt, environment call, or breakpoint).|
|**0x344**|`mip`|**Machine Interrupt Pending**: read-only indicator of pending interrupts.|
|**0xF14**|`mhartid`|**Hardware Thread ID**: read-only identifier of the current core (hardwired to zero for single core).|

While the `csr_file` module includes additional registers, the ones currently
functioning only as read/write scratchpads are omitted here for brevity. For
the complete architectural specification, refer to Chapter 2 and 3 of the
RISC-V Privileged Manual.

To ensure rapid PC redirections without introducing unnecessary latency, the
values of `mtvec` and `mepc` are continuously exposed to the datapath via
dedicated combinatorial outputs. For a detailed breakdown of how the core
interacts with these registers, refer to the page about [trap handling].

[trap handling]: traps/
